## 딥리서치 근거 요약 (2026-07-24 — 4축 병렬 리서치, 출처 하단)

**핵심 통찰: 우리는 "아웃링크" 서비스라 유저를 밖으로 내보낸다. 그래서 체류시간 기반 배너광고는 구조적으로 불리하고, "클릭·전환을 수익화"하는 모델(제휴 CPS, CPC 네이티브)이 유리하다.** 이 가설은 리서치로 검증됨 — 배너 CPC는 클릭당 20~30원인데 제휴는 전환당 수천~수만원. 그리고 우리의 개인화(취향 벡터)가 제휴 전환율을 끌어올린다(개인화 추천이 무작위 대비 전환율 유의미 상승 — 벤더 수치는 과장 감안해도 방향은 확실).

### 지금 당장 할 것 (P0 실행)
1. **쿠팡파트너스 제휴 카드** — 취향 벡터 상위 카테고리 → 상품 매칭, 피드에 네이티브 카드로. **반드시 "사용자가 직접 탭"하는 방식**(자동이동 = "납치광고", 2025년 쿠팡 형사고소 10곳). 카드에 "쿠팡 파트너스 활동으로 수수료를 받을 수 있음" 고지 배지 **상시 최상단** 노출(표시광고법: 제목/첫 부분 필수).
2. **편집 POV를 카드에 얹기** — "왜 핫한지 한 줄 요약"이 ①광고 심사 통과(스크랩 아님 소명) ②제휴 전환율 ③편집샵 가치를 **동시에** 올림. 수익화와 제품 품질이 같은 레버.

### 직접 경쟁자 jagei 실증 플레이북 (소스코드로 확인 — 가장 신뢰도 높은 레퍼런스)
jagei.co.kr HTML을 직접 분석한 결과, **우리가 하려는 것의 정답지가 이미 돌아가고 있음**:

## Problem Statement

내 취향 피드는 기능적으로 완성(테스트 58개 통과)되어 배포까지 마쳤지만 수익 구조가 없다. 서버·번역 API·유료 플랜(디스크 영속화) 등 운영 비용은 사용자 증가와 함께 커지는 반면 수입은 0이므로, 지속 운영이 불가능하다. 최우선 목표가 "합법 범위 내 수익화 가능한 프로덕트"인 만큼, 트래픽이 작을 때부터 작동하는 수익원을 먼저 붙이고 규모에 따라 층을 쌓는 설계가 필요하다.

## Goals

1. **소규모 트래픽에서도 첫 수익 발생** — 출시 후 30일 내 첫 제휴 수수료/광고 수익 기록 (금액 무관, 파이프라인 검증이 목적).
2. **취향 데이터를 수익 전환** — 추천엔진의 1st-party 취향 벡터로 제휴 카드 CTR ≥ 일반 배너 대비 2배 (자체 A/B 측정).
3. **비용 자립** — MAU 1천 시점에 월 인프라 비용(Render 유료 플랜 + 번역 API) 100% 커버.
4. **법적 안전 유지** — 광고·제휴 도입 후에도 legal.md 원칙(아웃링크·발췌 캡·본문 미복제) 위반 0건, 표시광고법 고지 100% 적용.

## Non-Goals

- **성인(19금) 영역 수익화** — 광고 네트워크 정책 위반 리스크가 크고 계정 정지 시 전체 수익원이 죽는다. 19금 뷰에는 광고·제휴 카드를 서빙하지 않는다.
- **콘텐츠 유료화(페이월)** — 아웃링크 모델 특성상 콘텐츠는 원저작자 것. 남의 콘텐츠에 과금하는 순간 법적 안전 모델이 깨진다. 과금 대상은 항상 "우리 기능"이다.
- **유저 데이터 판매/외부 광고 타게팅 제공** — 개인정보보호법 리스크. 취향 벡터는 서버 내부 랭킹에만 쓴다.
- **자체 광고 영업(직판)** — 트래픽 규모가 안 됨. 네트워크 광고로 시작.
- **네이티브 앱 전환** — PWA 유지 (앱스토어 수수료 30% 회피가 오히려 수익 이점).

## 수익원 우선순위

### P0 — 제휴 커머스 카드 (첫 수익원)

피드 카드 사이에 취향 기반 제휴 카드(쿠팡파트너스 등)를 네이티브 형태로 삽입.

- **왜 첫 번째인가**: 광고 네트워크와 달리 트래픽 심사 문턱이 없고, David가 이미 쿠팡파트너스 운영 경험·계정 보유. 추천엔진의 카테고리/태그 가중치를 그대로 상품 카테고리 매핑에 재사용 가능.
- **동작**: `engine`이 N개 피드 아이템마다 1개 제휴 슬롯 삽입(빈도 설정값). 유저 취향 벡터 상위 카테고리 → 제휴 상품 카테고리 매핑. 카드에는 `광고`/`파트너스` 라벨 + "이 포스팅은 쿠팡 파트너스 활동의 일환으로, 이에 따른 일정액의 수수료를 제공받습니다" 고지.
- **수용 기준**:
  - [ ] 제휴 카드는 일반 아이템과 명확히 구분되는 라벨 표시 (표시광고법)
  - [ ] 19금 뷰·미성년 추정 세션에는 미노출
  - [ ] 카드 클릭도 아웃링크 (제휴 링크로 이동, 프레이밍 없음)
  - [ ] 슬롯 빈도는 env로 조절 가능, 0이면 완전 비활성
  - [ ] 제휴 카드는 추천 학습 신호(skip/dwell)에서 제외 — 오학습 방지

### P0 — 광고 네트워크 (Kakao AdFit → AdSense)

- **왜 AdFit 먼저**: AdSense는 "짜깁기/스크랩 콘텐츠" 정책 심사에서 어그리게이터가 거절되기 쉽다. AdFit이 국내 심사 문턱이 낮고 모바일 웹 최적화. AdSense는 유저 생성 콘텐츠(내 공간 글·댓글) 비중이 쌓인 뒤 재도전.
- **수용 기준**:
  - [ ] 피드 내 광고 슬롯은 제휴 카드와 합산 빈도 상한 적용 (피드 10개당 광고+제휴 ≤ 2)
  - [ ] 19금 뷰 광고 미서빙
  - [ ] 광고로 인한 스크롤 성능 저하 없음 (lazy load)

### P1 — 프리미엄 구독 ("취향 프리미엄")

무료: 광고 포함 + 기본 기능 전부. 프리미엄(월 2,900~4,900원 밴드 테스트):

- 광고·제휴 카드 제거
- 해외 커뮤니티 **무제한 번역** (번역 API는 실비용 → 무료 티어는 일일 캡, 프리미엄 무제한. 비용-가격 정합)
- digest 푸시 고급 설정 (키워드 알림, 빈도 커스텀)
- 몰입 모드 고급 기능 (읽기 테마 등)
- **결제**: PWA 웹 결제 — 토스페이먼츠 or 카카오페이 정기결제. 앱스토어 수수료 없음.
- **선행 조건**: 유료 플랜 + `FEED_DB` 영속화 (결제 유저 데이터가 인메모리면 사고). 구독 상태 필드·만료 처리 필요.

### P2 — 미래 고려 (지금 안 함, 설계만 열어둠)

- **트렌드 리포트 B2B**: "지금 커뮤니티 화제" 집계 데이터의 뉴스레터/API 판매. 공개 신호 집계라 법적으로 안전한 편이나 수요 검증 전. `ingest.hotness` 데이터를 시계열로 쌓는 것만 미리 (스키마에 timestamp 유지).
- **커뮤니티 셀프서브 광고**: 소상공인이 특정 카테고리 피드에 직접 카드 게재. 관리자 콘솔 확장 필요.
- **후원(팁)**: 유저 글 작성자 후원. 결제 인프라 공유 가능.

## User Stories (요약)

- 무료 이용자: 피드 흐름을 깨지 않는 광고라면 무료로 계속 쓰고 싶다.
- 헤비 이용자: 광고 없이 + 해외 글 무제한 번역으로 보고 싶다 → 구독.
- 운영자(David): 슬롯 빈도·제휴 카테고리 매핑을 코드 수정 없이 관리자 콘솔에서 조절하고 수익 지표를 보고 싶다.

## 구현 상세 (Phase A — 슬롯 삽입 엔진, 2026-07-25)

Phase A(제휴 카드 슬롯 + 고지 + 19금 제외 + env 게이트)를 구현했다. `handoff.md` 절대원칙 1(더미 콘텐츠 금지)을 그대로 광고에 적용: **`COUPANG_PARTNER_ID`도 `AD_PREVIEW`도 없는 프로덕션 기본 실행은 제휴/광고 카드가 0개**다.

### 설계 원리 → 구현 매핑

| 심리·UX 원리 | 구현 |
|---|---|
| 동화(assimilation) + 명시(disclosure) | 슬롯 카드는 유기 카드와 동일한 `.card` 레이아웃·타이포. 단 상단에 대비색 "제휴"/"제휴 · 샘플" 배지(`ad-badge`), 하단에 쿠팡 법정 고지 문구를 항상 렌더 |
| 첫 화면 보호 | `AD_SKIP_FIRST`(기본 4) 이전엔 슬롯 없음 — `injectSlots`가 세션 전체 누적 위치(`cursor`)를 기준으로 판단해 페이지가 넘어가도 규칙이 끊기지 않음 |
| 딥존 가중(스크롤 중후반 배치) | 첫 슬롯이 `AD_SKIP_FIRST` 이후에만 등장 → 자연히 스크롤 초반보다 중후반에 배치됨. 이후 `AD_EVERY`(기본 9) 간격으로 반복 |
| 세션 캡 | `AD_MAX_PER_PAGE`(기본 2) — 한 페이지(요청)당 슬롯 상한. 간격 계산상 더 넣을 자리가 있어도 여기서 멈춤 |
| 적응형 밀도 | 유저별 제휴 클릭 반응성(관측 CTR ÷ `AD_BASELINE_CTR`)이 표본(`minSample=5`) 이상 쌓이면 `AD_EVERY`를 ±조정. 신호 부족 시 기본값 그대로(`adaptiveEvery(x, null) === x`) |
| 관련성 게이팅 | 취향벡터의 top 카테고리 학습 가중치(`recommender.js`의 `WEIGHT_CLAMP=6` 스케일)를 relevance로 변환해 `AD_MIN_RELEVANCE`(기본 0.3) 미달이면 해당 슬롯을 **비운다**(강제 채움 없음). 취향 신호가 전혀 없는 콜드스타트 유저는 후보 자체가 0개 |
| 앵커링 | 정가/할인가 병기(`priceOriginal`/`priceSale`) + 할인율. 실제 가격쌍이 없으면 아예 렌더하지 않음(가짜 할인 금지) |
| 다크패턴 금지 | 카운트다운·가짜 구매자수 카운터 없음. 클릭 전엔 완전히 정적. 클릭 시 항상 새 탭(`noopener,noreferrer`), 자동이동 없음("납치광고" 회피) |
| 19금/정치/종교 미노출 | `engine.js`의 `getFeed`가 `allowAdult`·`showTopics`(politics/religion)를 이미 계산해두므로, 그 상태를 그대로 재사용해 슬롯 삽입 자체를 건너뜀 |

### 신규/변경 파일

- **`src/feed/monetize.js`** (신규) — 슬롯 삽입 순수 함수 `injectSlots`, 적응형 밀도 `adaptiveEvery`/`adResponsivenessRatio`, A/B `assignVariant`/`applyVariant`, 후보 생성 `pickAffiliateCandidates`/`sampleAffiliateCandidates`, 슬롯 아이템 셰이핑 `makeSlotItem`. env 튜너블: `AD_EVERY`, `AD_SKIP_FIRST`, `AD_MAX_PER_PAGE`, `AD_MIN_RELEVANCE`, `AD_BASELINE_CTR`, `AD_AB`, `AD_PREVIEW`, `COUPANG_PARTNER_ID`.
- **`src/feed/engine.js`** — `getFeed`에 `_monetize()` 훅 추가. seen/exposure 기록 **이후**에 슬롯을 끼워 넣어 슬롯 아이템이 개인화 학습·중복제거 로직을 절대 오염시키지 않게 함. `nextCursor`는 유기 아이템 개수만 기준(슬롯 유무와 무관하게 페이지네이션 안정).
- **`src/feed/store.js`** — `recordAdEvent`(노출/클릭 카운터 + 원시 이벤트 로그, 최대 2000건), `adResponsiveness`(적응형 밀도 입력), `adminAdStats`(관리자 지표). `_persist`/`_load`에 `adEvents` 포함.
- **`src/feed/server.js`** — `POST /api/ad-signal`(노출/클릭 로깅, 엔진 아이템 풀 조회 없이 스토어에 직접 기록 — 슬롯 아이템은 풀에 없음), `GET /api/config`에 `monetization.enabled` 플래그, `GET /api/admin/stats`에 `ads` 필드 추가.
- **`src/feed/public/index.html`** — `appendAdCard`(전용 카드 렌더러: 배지·가격·이유·고지, 좋아요/스크랩 버튼 없음), `adImpressionObserver`(IntersectionObserver 기반 노출 로깅, threshold 0.5), 클릭 시 `ad-signal` click 로깅 후 새 탭 이동, 앱 전역 1회 고지 배너(`maybeShowAdNotice`, `feed_ad_notice_seen` localStorage 게이트).

### 1차지표/가드레일 데이터

`/api/ad-signal`이 노출·클릭을 유저별 카운터(`adStats`)와 원시 이벤트 로그(`adEvents`, userId/itemId/type/variant/timestamp)로 남긴다. 1차지표(세션당 수익 proxy = 클릭수)는 `adminAdStats().clicks`로 바로 집계된다. 가드레일("직후 유기카드 스크롤 지속 여부")은 이 원시 로그를 기존 `user.seen` 타임스탬프와 조인해 분석하는 것을 전제로 설계했다 — **이번 구현은 조인 없이 원시 데이터 수집까지만**이며, 자동 이탈률 계산은 후속 과제로 남겨둔다(과다구현 방지, WRC 효율 원칙).

### 알려진 제약 / 후속 과제

- 실제 쿠팡파트너스 상품 피드 연동은 미구현(약관 원문 확인이 David 확인 대기 — Open Questions 기존 항목 유지). `COUPANG_PARTNER_ID`를 설정해도 `opts.productFeed`를 별도로 넘기지 않으면 여전히 0개 — 반쪽짜리 크레덴셜이 카드를 만들어내지 않도록 의도적으로 막아둠.
- 광고 CPC 네트워크(AdFit, P0 두 번째 항목)는 아직 미착수. `kind: "ad"`를 위한 필드는 마련해뒀지만 후보 생성기는 없음.
- 리텐션 가드레일의 실제 이탈률 계산(로그 조인·집계)은 후속 과제.
