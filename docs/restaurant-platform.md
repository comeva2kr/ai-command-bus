# 맛집 통합 커뮤니티 (Restaurant Discovery Platform)

여러 소스(SNS·유튜브·네이버·커뮤니티)에서 맛집 언급을 모아 **광고/협찬을 자동
제외**하고, 교차검증된 맛집만 남긴 뒤, **위치 + 생활밀착 다중조건**으로 필터링해
보여주는 검색 엔진과 웹 UI입니다. 의존성 없이 순수 Node로 동작합니다.

## 실행

```bash
npm test          # 엔진 + 3가지 예시 쿼리 검증
npm run eats      # http://localhost:4173 웹 UI + API
```

## 파이프라인

```text
raw review-level signals (SNS·유튜브·네이버·커뮤니티)
  → authenticity    (★찐맛집 판별: 광고 제외 + 어뷰징/버스트 veto + 6요소 점수화)
  → geo gate        (반경 / 이동시간 / "OO 근처" 랜드마크)
  → multi-filter    (스타일·음식·메뉴·태그·편의·가격 다중 AND)
  → rank            (진위도·평점·선호일치·근접도 블렌딩)
```

| 모듈 | 역할 |
| --- | --- |
| **`src/restaurants/authenticity.js`** | **찐맛집 판별 엔진** — 리뷰+행동 신호로 7요소 점수화 + 광고 페널티 + 다중 veto → `authenticityScore`(0–100), `verdict`, `breakdown`, `reasons` |
| **`src/restaurants/corpus.js`** | **코퍼스 그래프 분석** — 교차업소 리뷰링/락스텝 담합 탐지(CopyCatch/FRAUDAR 원리)·싱글톤·작성자 통계, 위장 저항 |
| **`src/restaurants/beliefprop.js`** | **믿음 전파(LBP)** — 리뷰어↔업소 그래프에 사기 확률 전파(FraudEagle/SpEagle), 네트워크 신호 통합 |
| `src/restaurants/textintegrity.js` | 근접중복(셰인글링·Jaccard) + AI 생성 리뷰 탐지 (텍스트 있을 때) |
| `src/restaurants/timeseries.js` | 시계열 평점 급등 이상탐지 (Xie, 볼륨·평점 상관 스파이크) |
| `src/restaurants/ingest.js` | 배치 래퍼 — 코퍼스 분석 + 믿음 전파 후 각 업소를 컨텍스트와 함께 판별, 미검증 제외(기본) |
| `src/restaurants/geo.js` | 하버사인 거리, 이동시간 추정, `travelBudgetToRadiusKm`(차로 N분→반경), 랜드마크 좌표 |
| `src/restaurants/taxonomy.js` | 스타일·음식·태그·**메뉴·메뉴특성** 동의어 정규화(`고기집→고깃집`, `살아있는→활`) |
| `src/restaurants/filter.js` | 하드조건(제외) + 소프트선호(가점) 다중조건 필터 |
| `src/restaurants/query.js` | 검증→지오→필터→랭킹 결합 `search()` |
| `src/restaurants/server.js` | 무의존성 HTTP API + 정적 프론트엔드 |

## ⭐ 핵심: 찐맛집 판별 알고리즘 (`authenticity.js`)

이 서비스의 본질은 UI가 아니라 **다양한 소스에서 진짜 맛집과 광고로 띄운 곳을
가려내는 판별 알고리즘**입니다. 속성(고깃집·키즈카페 등)이 아무리 잘 맞아도,
소수 계정이 오픈발로 별5를 도배한 곳은 걸러져야 합니다.

소스가 많아질수록(틱톡·유튜브 쇼츠·인스타/릴스·블로그·커뮤니티·네이버/다음/카카오
지도·캐치테이블·망고플레이트·다이닝코드 등) **말(리뷰)은 조작이 쉬워집니다.** 그래서
엔진은 두 축으로 진짜를 붙잡습니다:

1. **소스 클래스 신뢰 + 교차 확증** — 각 플랫폼을 `short_video / social / map / app /
   community` 클래스로 묶고 클래스별 신뢰가중치를 다르게 줍니다. 숏폼은 도달은 크나
   저신뢰(0.5), 커뮤니티·지도·앱은 고신뢰. **서로 다른 독립 클래스에서 확증**돼야
   진짜로 인정합니다.
2. **행동 데이터(revealed preference)** — 예약·웨이팅·재방문율·저장수. *말이 아니라
   실제 행동*이라 조작이 가장 어렵고 가중치가 가장 큽니다.

엔진은 리뷰 단위 신호 + 행동 데이터를 받아 7개 요소를 0~100으로 점수화하고,
가중합에 광고 페널티를 곱한 뒤, 강한 가짜 신호에는 상한(veto)을 씌웁니다. 모든
하위 점수와 근거가 설명가능(explainable)하게 반환됩니다.

| 요소 | 가중 | 진짜 신호 | 가짜 신호 |
| --- | --- | --- | --- |
| `behavior` 행동 데이터 | 0.22 | 예약·웨이팅·재방문·저장 (실제 방문) | 말만 많고 안 감 |
| `diversity` 작성자 다양성 | 0.18 | 독립 계정 다수 (HHI 낮음) | 소수 계정 도배 → **어뷰징 veto** |
| `local` 로컬·재방문 | 0.17 | 동네사람·단골·재방문 | 인플루언서 원정만 |
| `classBreadth` 소스 클래스 교차 | 0.13 | 지도+커뮤니티+앱 등 다클래스 | 숏폼 한 클래스만 |
| `sustain` 시간 지속성 | 0.12 | 수년간 꾸준, 노포 보너스 | 단기 폭발 후 소멸 |
| `realism` 평점 현실성 | 0.10 | 자연 분포·비판 리뷰 존재 | 전부 별5, 흠 없음 |
| `texture` 후기 구체성 | 0.08 | 특정 메뉴·단점 서술 | 뻔한 홍보 문구 |

**광고 페널티**: `score *= (1 − 0.5 × 광고비율)`, 협찬 리뷰는 계산 전 제외.

**Veto(상한) 규칙** — 속성이 맞아도 가짜면 통과 불가:
- 작성자 편중(고유작성자/후기수 < 0.4 또는 HHI > 0.35) → `어뷰징의심`, 상한 35
- 광고비율 > 0.6 → `광고의심`, 상한 28
- **숏폼 비중 ≥ 55%인데 지도·앱·커뮤니티 확증 없고 행동 데이터 빈약** → `바이럴거품`, 상한 40
- 유기 표본 < 6 또는 단일 소스 클래스 → `정보부족`
- 30일 내 70%↑ 몰림 + 짧은 기간 → 단기폭발 감점

**판정 등급**: `찐맛집(≥80)` · `검증됨(≥65)` · `보통(≥45)` · `미검증/광고의심/어뷰징의심/바이럴거품`.
기본 검색은 `verified`(찐맛집·검증됨)만 노출합니다. (`includeUnverified: true`로 전체 조회)

**판별 예시** (시드 데이터) — 셋 다 속성 필터는 통과하지만 가짜 유형별로 걸러집니다:

| 업소 | 점수 | 판정 | 이유 |
| --- | --- | --- | --- |
| R-101 성수 뒷고기집 | 84 | 찐맛집 | 4클래스 교차·작성자26명·2년·재방문42%·예약46건/주 |
| R-104 오픈발 신상 고깃집 | 28 | 어뷰징의심 | 속성 완벽 일치하나 3계정이 2주간 별5 도배 |
| R-103 협찬 고깃집 | 22 | 광고의심 | 광고비율 70% |
| R-105 쇼츠 바이럴 파스타 | 35 | 바이럴거품 | 숏폼에서만 폭발, 지도·행동 확증 전무(재방문 6%) |

태그 매칭이 아니라 **진위 판별**이 결과를 결정합니다.

### 적용한 연구 기반 기법 (research-applied)

가짜 리뷰/어뷰징 판별의 학계·업계 연구를 리서치해 조작 저항성이 높은 순으로 반영했습니다.

| 기법 | 출처 | 우리 엔진에 적용 |
| --- | --- | --- |
| **행동 신호 > 언어 신호** — 예약·재방문 등 행동 데이터가 텍스트보다 강력(Yelp 필터는 행동 기반, 언어 단독 55–68% vs 행동 결합 ~89%) | Mukherjee et al., *What Yelp Fake Review Filter Might Be Doing?*, ICWSM 2013 | `behavior` 요소에 최대 가중(0.22) |
| **리뷰링/락스텝** — 소수 계정이 여러 업소를 같은 시간대에 함께 리뷰. **위장(camouflage) 저항성** — 정상 리뷰를 섞어도 담합 증거는 희석되지 않음 | Beutel et al., *CopyCatch*, WWW 2013 · Hooi et al., *FRAUDAR*, KDD 2016 | `corpus.js` 코퍼스 그래프에서 교차업소 락스텝 탐지 → `담합의심` veto |
| **버스트 + 싱글톤 + 평점 상관** — 스팸 공격은 짧은 기간 별점 급등과 상관 | Xie et al., *Review Spam Detection via Temporal Pattern Discovery*, KDD 2012 · Fei et al. co-bursting | `조직적버스트` veto (버스트+별5도배+낮은 다양성) |
| **평점 분포 형태** — 진짜는 J커브, 가짜는 "중간 없음(별5 도배)"·U자(양극단) | Amazon/TripAdvisor 분포 연구, bimodal/co-bursting 문헌 | `realism`을 분포 형태 기반으로 업그레이드 |
| **작성자 편중(HHI)** — 소수 계정 집중 | 스팸 그룹 탐지 문헌 | `작성자편중` veto |
| **언어 기만 신호** — LIWC·구체성; 단독은 약해 보조 지표로만 | Ott et al., *Finding Deceptive Opinion Spam*, ACL 2011 | `texture`(구체성) 보조 가중(0.08) |

핵심 원칙(FRAUDAR): 가짜는 **정상 리뷰를 섞어 위장**할 수 있으므로, 위장에도 줄어들지
않는 신호(교차업소 락스텝, 행동 데이터)에 신뢰를 앵커링합니다. 데모의 R-106이 그 예 —
겉보기 J커브·작성자 다양성으로 단일 업소 관점에선 깨끗해 보이나, 코퍼스 리뷰링 탐지가
잡아냅니다.

### 슈퍼파워 5종 (심화 적용)

추천 순서대로 연구 기법을 추가 구현했습니다. 각각 다른 가짜 유형을 잡습니다.

| # | 기법 | 출처 | 데모가 잡는 가짜 |
| --- | --- | --- | --- |
| ① | **싱글톤 계정 공격** — 일회성 신규계정 폭주(전역 리뷰 1건) + 버스트 + 평점 편향 | Xie et al., KDD'12 | R-107: 작성자 20명(다양성 통과)이지만 전부 일회성 → `싱글톤공격` |
| ② | **근접중복/템플릿** — 셰인글링 3-gram + Jaccard≥0.8 군집 | 리뷰 스팸 문헌 | R-108: 거의 동일 문장 복붙 → `복붙리뷰` |
| ③ | **AI 생성 탐지** — 업스트림 탐지기 확률 소비 + 휴리스틱(뻔한 표현·구체성 결여) | Ott et al. ACL'11 외 | R-109: 서로 다르지만 AI스러운 문장 → `AI리뷰의심` |
| ④ | **믿음 전파(LBP)** — 리뷰어↔업소 MRF에 사기 확률 전파, 신호 통합 | Akoglu et al. FraudEagle ICWSM'13 · Rayana & Akoglu SpEagle KDD'15 | R-104: 밀집 담합 블록 네트워크 확률 0.99 → `네트워크사기전파` |
| ⑤ | **시계열 평점 급등** — 30일 버킷 볼륨 스파이크 + 평점 리프트 상관 | Xie et al., KDD'12 | R-111: 오래된 정상 이력에 최근 별5 급조(전체 버스트 검사는 통과) → `평점급등의심` |

> 정직한 한계: 순수 구조적 LBP는 위장(honest 리뷰 혼입)에 취약합니다 — 바로 그래서
> FRAUDAR 원리의 **위장-저항 리뷰링 탐지(`corpus.js`)를 함께** 돌려 방어를 이중화합니다.
> LBP는 SpEagle처럼 메타데이터 로컬 사전확률을 시드로 그래프 전체에 신호를 전파·통합하는
> 층입니다. 실서비스 규모에선 리뷰링 페어 비교를 FRAUDAR류 밀집블록 탐색으로 대체합니다.

이제 엔진은 **9가지 가짜 유형**을 판별합니다: 광고의심 · 어뷰징의심 · 바이럴거품 ·
담합의심 · 조직적버스트 · 싱글톤공격 · 복붙리뷰 · AI리뷰의심 · 평점급등의심.

## 위치 필터

- 내 좌표(`location: {lat,lng}`) 또는 `location: {near: "세종"}` 랜드마크 기준
- `radiusKm` 직접 지정, 또는 `travel: {mode:"car", minutes:30}` → 반경 자동 변환
- 결과마다 `distanceKm` / `travelMinutes` 표시
- 이동시간은 블렌딩된 평균속도 모델(도심+간선)입니다. 실서비스에서는 라우팅/ETA API로 교체하는 지점입니다.

## 다중조건 필터

| 필드 | 의미 |
| --- | --- |
| `styles` / `cuisines` | 장소 스타일 / 음식 종류 |
| `menu` + `menuAttrs` | 메뉴 단위 검색 (예: `고등어회` + `["활"]` = 살아있는 고등어회) |
| `tagsAll` / `tagsAny` / `excludeTags` | 분위기·특성 태그 (필수 / 하나이상 / 제외) |
| `require` | 필수 편의 (`{kidsCafe:true, partition:true}`) |
| `excludeFranchise` | 프랜차이즈 제외 |
| `priceMin`~`priceMax` | 가격대(₩1–4) |
| `prefer` / `preferFeatures` | 소프트 선호 — 있으면 순위 가점 |

## 예시 쿼리 (프리셋 제공)

**A. 프랜차이즈 아닌 뒷고기 고깃집 + 키즈카페**
```json
{ "styles":["고깃집"], "cuisines":["돼지고기"], "excludeFranchise":true,
  "require":{"kidsCafe":true}, "prefer":["뒷고기"] }
```

**B. 이자카야 · 숙성회 · 싼 분위기 아님 · 파티션 선호**
```json
{ "styles":["이자카야"], "tagsAll":["숙성회"], "excludeTags":["가성비"],
  "prefer":["고급스러운"], "preferFeatures":{"partition":true}, "priceMin":2 }
```

**C. 세종 인근 · 차로 30분 · 활(살아있는)고등어회**
```json
{ "location":{"near":"세종"}, "travel":{"mode":"car","minutes":30},
  "menu":"고등어회", "menuAttrs":["활"] }
```

## API

- `GET /api/meta` — 랜드마크·스타일·음식·태그·메뉴·편의·프리셋 목록
- `POST /api/search` — 본문에 쿼리 JSON → `{ meta, results }`
- `GET /api/preset?name=kids-pork|izakaya-aged|sejong-mackerel`

## 데이터

`src/restaurants/data/seed.js`는 리뷰 단위 신호(작성자·시점·별점·협찬·로컬·재방문·
구체성)를 담은 샘플이며, `makeReviews`가 압축 스펙을 결정적으로 전개합니다. 실제
서비스에서는 이 자리에 각 플랫폼 커넥터(네이버 플레이스/블로그, 유튜브,
인스타그램, 지역 커뮤니티)가 수집한 실제 리뷰 스트림이 들어갑니다.
